# ТЗ: BPM-система согласования документов

> Внутреннее техническое задание. Веб-решение, не 1С.
> Версия 1.0 · Контекст: Казахстан, Astana (UTC+5)

---

## 1. Назначение

Система электронного согласования внутренних документов организации:
служебные записки, заявки на договор, заявки на оплату и др.

Маршрут согласования вычисляется динамически из **матрицы маршрутов**
в разрезе: вид объекта × организация × проект. Исполнители определяются
по **должностям** (должность = роль), получаемым из 1С:ЗУП.

### Не входит в объём

- Электронная цифровая подпись (рассмотреть отдельно)
- Мобильное приложение (только адаптивный веб)
- Полноценный документооборот с номенклатурой дел и архивным хранением
- BPMN-моделирование (маршруты задаются матрицей, не диаграммами)

## 2. Ландшафт систем

```
┌─────────────┐
│  1С:ЗУП     │──── сотрудники, должности, ──────┐
│             │     подразделения, отсутствия     │
└─────────────┘                                   ▼
                                          ┌───────────────┐
┌─────────────┐   организации, проекты,   │               │
│ 1С:Бухгал-  │──── контрагенты ─────────▶│      BPM      │
│   терия     │                            │   (веб)       │
│             │◀─── контрагенты, договоры, │               │
└─────────────┘     история согласования   └───────────────┘
                                                   ▲
┌─────────────┐                                    │
│  AD / LDAP  │──── аутентификация ────────────────┘
└─────────────┘
```

### Разделение ответственности

| Данные | Источник истины | Направление |
|---|---|---|
| Сотрудники, должности, подразделения, отсутствия | ЗУП | ЗУП → BPM |
| Организации, проекты | Бухгалтерия | БУХ → BPM |
| Контрагенты (существующие) | Бухгалтерия | БУХ → BPM |
| Контрагенты (новые) | BPM | BPM → БУХ |
| Договоры | BPM | BPM → БУХ |
| Маршруты, задачи, визы, история | BPM | внутреннее + история → БУХ |
| Учётные записи | AD | AD → BPM |

**Правило:** BPM никогда не редактирует данные систем-источников.
Только чтение справочников и создание новых объектов (контрагент, договор).

## 3. Аутентификация и авторизация

### 3.1 Схема

```
Пользователь → BPM → Keycloak (OIDC) → LDAP/AD
```

Keycloak как посредник. Прямой LDAP-bind из приложения допустим только
для внутреннего контура без внешнего доступа.

Kerberos/SPNEGO (вход без пароля в домене) — опционально, вне первой версии.

### 3.2 Связка AD ↔ ЗУП

Критическая задача: в AD нет ИИН, в ЗУП нет `sAMAccountName`.

Порядок сопоставления:

1. **Общий атрибут** — `employeeID` в AD = табельный номер или ИИН в ЗУП
2. **Email** — `mail` в AD = корпоративная почта в ЗУП
3. **Ручной маппинг** — админ-экран, обязателен как fallback

**Запрещено сопоставлять по ФИО.** Однофамильцы, отчества, «ё»,
двойные пробелы — гарантированные ошибки.

> **Первое действие проекта:** выгрузить AD в CSV, выгрузить ЗУП в CSV,
> посчитать процент автоматического сопоставления по выбранному атрибуту.
> Если < 80% — закладывать время на чистку данных в AD.

### 3.3 Синхронизация с AD

- Импорт по расписанию (раз в час) из **целевых OU** — фильтр обязателен,
  иначе импортируются сервисные учётки, компьютеры, ящики
- `objectGUID` как ключ — **не** `distinguishedName` (меняется при переводе
  между OU) и **не** `sAMAccountName` (меняется при смене фамилии)
- Отключённые в AD (`userAccountControl` бит 2) → `status = DISABLED`
- Новые в AD → создаются без привязки к сотруднику, в очередь сопоставления
- Из BPM в AD не пишем ничего

### 3.4 Двойная деактивация

Два независимых признака, проверяются **оба**:

| Признак | Следствие |
|---|---|
| Учётка отключена в AD | Вход невозможен |
| Сотрудник уволен в ЗУП | Вход возможен, задачи не назначаются, исключён из матрицы |

Кейсы: уволенный с активной AD-учёткой (доработка периода) не должен
получать новые согласования; отключённый в AD, но числящийся в ЗУП
(декрет, длительная командировка) — тоже.

### 3.5 Несопоставленный пользователь

Вошёл через AD, связи с сотрудником ЗУП нет:

- Вход разрешён, интерфейс доступен
- Явное сообщение: «Учётная запись не сопоставлена с сотрудником,
  обратитесь к администратору»
- Создание заявок заблокировано (нет организации/подразделения/руководителя)
- Уведомление администратору

Молчаливый вход в пустой интерфейс — источник обращений в поддержку.

### 3.6 Роли

**Роли хранятся в BPM, не в группах AD.** AD отвечает «кто ты»,
BPM — «что тебе можно». Исключение: группа `BPM_Admins` в AD допустима.

| Роль | Права |
|---|---|
| Инициатор | Создаёт заявки, видит свои |
| Согласующий | Видит процессы со своей задачей (текущей или прошлой) |
| Наблюдатель | Видит все процессы своей организации, без права действий |
| Настройщик матрицы | Правит маршруты, не участвует в согласовании |
| Администратор | Всё + принудительное завершение + разбор обменов |

Бухгалтеры → «Наблюдатель». **Роль не должна требовать записи в матрице** —
иначе настройщик, добавляя бухгалтера ради видимости, вставит его
согласующим.

Ограничение по организации обязательно: наблюдатель ТОО «А» не видит
документы ТОО «Б». Организация берётся из сопоставления с ЗУП.

### 3.7 Технические требования

- **LDAPS (порт 636)**, сертификат домена в trust store приложения
- **Сервисная УЗ** только на чтение, пароль не истекает, в Vault
  или переменных окружения — не в конфиге репозитория
- **Два контроллера домена** в конфигурации — единственный DC = SPOF
- **Paged Results Control** обязателен: при > 1000 записей AD вернёт
  частичный результат без ошибки
- **Кэш сессии** — обращение в AD при входе и обновлении токена, не на
  каждый запрос

## 4. Матрица маршрутов

### 4.1 Структура

**Измерения (условия отбора):**

- Вид объекта (служебная записка / заявка на договор / …)
- Организация (NULL = любая)
- Проект (NULL = любой)
- Номер этапа
- Порядок в этапе (для параллельных)

**Ресурсы:**

- Способ адресации (`resolver_type`)
- Должность (для фиксированной адресации)
- Тип этапа: последовательно / параллельно / кворум
- Срок (часы)
- Обязательность: обязательно / опционально / пропустить если нет исполнителя
- Приоритет (число, меньше = выше)
- Период действия (`valid_from` / `valid_to`)

Одна строка = один участник одного этапа.
Маршрут = выборка строк по контексту, сортировка по номеру этапа.

**Пустое значение измерения = «любой».** Без этого матрица разрастётся
до тысяч строк и станет несопровождаемой.

### 4.2 Способы адресации (`resolver_type`)

| Значение | Разрешение |
|---|---|
| `POSITION_IN_ORG` | Должность + организация → `employments` |
| `POSITION_IN_PROJECT` | Должность + проект → `project_assignments` |
| `INITIATOR` | Сам инициатор |
| `INITIATOR_MANAGER` | Вверх по иерархии подразделений до руководящей должности |
| `PROJECT_MANAGER` | РП текущего проекта |

Без вычисляемых способов адресации придётся заводить строки матрицы
под каждое подразделение.

### 4.3 Разрешение конфликтов

Когда подходит несколько строк (правило «для всех организаций»
и правило «для ТОО А по проекту 15»):

**Явный приоритет** — реквизит `priority`, побеждает наименьшее число.
Предзаполнение автоматическое по специфичности (чем меньше NULL
в измерениях, тем выше приоритет), но с возможностью ручной правки.

Модель — **замещение**, не дополнение. Побеждает одна группа правил,
а не суммируются несколько. Аддитивная модель гибче, но резко усложняет
отладку.

### 4.4 Тестировщик матрицы

**Обязательный экран.** Пользователь задаёт контекст (вид объекта,
организация, проект, инициатор) и видит рассчитанный маршрут
с конкретными ФИО — до запуска реального процесса.

Без него настройщики работают вслепую.

### 4.5 Версионирование

- Матрица версионируется через `valid_from` / `valid_to`
- При запуске процесса рассчитанный маршрут сохраняется в
  `process_instances.route_snapshot` (JSONB)
- **Изменение матрицы не влияет на запущенные процессы**

## 5. Разрешение исполнителя

### 5.1 Алгоритм

```
1. Отобрать правила:
     object_type = X
     AND (organization_id = Y OR organization_id IS NULL)
     AND (project_id = Z OR project_id IS NULL)
     AND CURRENT_DATE BETWEEN valid_from AND valid_to
   Сортировка по priority, отбор победившей группы

2. Для каждой строки — разрешить исполнителя по resolver_type

3. Проверить absences (отпуск/больничный) и substitutions
   → подменить на заместителя, сохранив substitute_for_id

4. Проверить статус: активен в AD И не уволен в ЗУП

5. Не найден исполнитель → ЗАБЛОКИРОВАТЬ запуск с сообщением
   «Не определён исполнитель для должности X по организации Y»

6. Сохранить route_snapshot
```

Шаг 5 критичен: молчаливый пропуск этапа означает, что процесс уйдёт
без нужной визы, и это вскроется через месяц.

### 5.2 Крайние случаи

**Одна должность — несколько человек** (два юриста в организации).
По умолчанию: задача обоим, любой закрывает. Признак «Основной»
в `employments` — опционально.

**Совпадение исполнителей** (инициатор оказался согласующим на этапе 3;
один человек занимает две должности в маршруте).
Правило: автоматически закрывать такие задачи с пометкой
«Совпадение с инициатором/предыдущим этапом». Иначе люди дважды
кликают одно и то же.

**Совместители** — один сотрудник, несколько записей `employments`
с разными должностями. Матрица может найти его дважды → см. выше.

## 6. Модель процесса

### 6.1 Типы этапов

- Последовательный
- Параллельный: все обязаны / любой достаточно / кворум N из M
- Обязательный / опциональный

### 6.2 Статусы процесса

```
DRAFT              — черновик у инициатора
IN_PROGRESS        — идёт согласование
REJECTED           — отклонён
APPROVED           — согласован
PENDING_EXPORT     — согласован, ожидает выгрузки в БУХ
EXPORTED           — выгружен в БУХ
CANCELLED          — отозван инициатором
FORCE_CLOSED       — принудительно закрыт администратором
```

`PENDING_EXPORT` — отдельный статус, а не подсостояние `APPROVED`.
Иначе сценарий «согласование прошло, отправка упала» останется невидимым
и документ потеряется в тишине.

### 6.3 Обработка отклонения

Самая недооценённая часть. Решение фиксируется до начала разработки.

**Принято:** возврат инициатору на доработку с **сохранением ранее
полученных виз**. При повторной отправке маршрут пересчитывается,
уже полученные визы на неизменившихся этапах сохраняются, если
документ не менялся в части, относящейся к этапу.

Упрощённый вариант для v1: полный сброс виз при возврате.
Реализовать частичное сохранение во второй итерации.

### 6.4 Обязательная механика

- **Эскалация по срокам** — просрочка → уведомление руководителю исполнителя
- **Отзыв процесса** инициатором до финальной визы
- **Принудительное завершение** администратором с обязательным комментарием
- **Блокировка изменения документа** во время согласования
  (или версионирование с перезапуском виз)
- **Полный аудит**: кто, когда, что, комментарий, IP
- **Возврат из БУХ** — кнопка у бухгалтера, процесс переоткрывается
  на нужном этапе, а не создаётся заново

### 6.5 Метрики (закладывать сразу)

- Среднее время прохождения этапа
- Узкие места (этапы с максимальным временем)
- Доля отклонений по согласующим
- Просрочки по должностям и организациям

История хранится структурированно, не текстом — иначе аналитики не будет.

## 7. Интеграция с 1С:ЗУП

### 7.1 Транспорт

HTTP-сервис на стороне ЗУП (расширение конфигурации), JSON, пагинация,
дельта по дате изменения. Опрос BPM по расписанию (15–60 мин) +
кнопка ручной синхронизации.

Альтернативы:
- **OData ЗУП** — не требует доработки конфигурации, но медленный
  и неудобный для составных выборок
- **Очередь (RabbitMQ)** с публикацией по подписке на проведение
  кадровых документов — правильнее, но больше работ на стороне 1С;
  оправдано при большом штате

### 7.2 Состав данных

```
Должности:        id(guid), наименование
Подразделения:    id(guid), наименование, родитель, организация
Сотрудники:       id(guid), ФИО, email, статус
Кадровые данные:  сотрудник, организация, подразделение, должность,
                  дата начала, дата окончания, вид занятости
Отсутствия:       сотрудник, период, вид (отпуск / больничный)
```

### 7.3 Правила

- **Ключ — GUID из 1С.** Никогда не ФИО, не табельный номер
- **Удалений не бывает** — ЗУП увольняет, не удаляет. Синхронизация =
  upsert со статусом, не полная перезапись
- **Уволенный остаётся в БД навсегда** — на нём висит история согласований
- **Дельта-синхронизация** обязательна: запрос изменений с последней
  метки времени. Полный слепок при 5000 сотрудников уже неприятен
- **Email в ЗУП часто пуст или устарел.** Источник email для уведомлений —
  AD, не ЗУП. ЗУП-email используется только как fallback

## 8. Интеграция с 1С:Бухгалтерия

### 8.1 Входящее (БУХ → BPM)

HTTP-сервис на стороне БУХ, JSON, пагинация, дельта.
Справочники: организации, проекты, контрагенты.

> **Проверить до разработки:** ведутся ли проекты в БУХ как отдельный
> справочник. Часто это субконто, направление деятельности или
> подразделение — от этого зависит формат endpoint.

### 8.2 Исходящее (BPM → БУХ)

**Модель обмена — Pull** (рекомендуется в 1С-контурах):

```
BPM выставляет:  GET  /api/outbox/pending
                 POST /api/outbox/{id}/ack
1С забирает регламентным заданием и подтверждает
```

Альтернатива — **Push** (BPM вызывает HTTP-сервис БУХ): проще,
но 1С бывает недоступна (обновление, регламентные работы),
нужны ретраи с backoff.

**Выбор зависит от того, доступна ли Бухгалтерия по сети из BPM.**
→ уточнить перед стартом.

### 8.3 Outbox-паттерн

Обязателен в любом варианте:

- Таблица `integration_outbox` со статусами и счётчиком попыток
- Отдельный воркер
- **Админ-экран «зависшие обмены»** — без него разбор инцидентов
  превратится в чтение логов
- Кнопка «Повторить отправку», доступная бухгалтеру

### 8.4 Идемпотентность

Каждый исходящий запрос с уникальным `idempotency_key` (UUID).
Повтор с тем же ключом не создаёт второй объект.

На стороне 1С: регистр сведений «ОбработанныеЗапросы»:
`ключ → ссылка на созданный объект`.

Без этого при таймауте/ретрае — двойники, которые бухгалтерия
вычищает вручную.

### 8.5 Контрагенты

Порядок операций строгий: **контрагент → договор**.

**Статусы:**

```
DRAFT     — создан в заявке, не согласован
PENDING   — отправлен в БУХ, ответа нет
SYNCED    — создан в БУХ, получен external_id
FAILED    — БУХ отклонил (дубль, ошибка реквизитов)
IMPORTED  — пришёл из БУХ
```

Договор не отправляется, пока контрагент не `SYNCED`.

**Дедупликация:**

- Поиск по **БИН/ИИН**, не по наименованию («ТОО Альфа», «ТОО "Альфа"»,
  «Альфа, ТОО» — три разных строки)
- БИН нормализовать: убрать пробелы, строка фиксированной длины,
  **не сравнивать как число** (теряются ведущие нули)
- Исторические контрагенты в БУХ часто без БИН → fallback на ручной
  выбор из списка похожих
- **Дедупликация контрагентов — с ручным подтверждением бухгалтера**,
  не автоматическая

**Коллизия:** контрагента с тем же БИН завели вручную в БУХ, пока шло
согласование. БУХ возвращает **не ошибку, а существующую ссылку** —
BPM записывает её и переходит в `SYNCED`.

### 8.6 Сопоставление организаций

Организация имеет **разные GUID в ЗУП и в Бухгалтерии** (если базы
разворачивались независимо — GUID не совпадут никогда).

Алгоритм:

```
1. Поиск в external_mapping (system=БУХ, external_id=guid)
2. Не нашли → поиск по БИН среди сопоставленных организаций
3. Нашли по БИН → создать запись в external_mapping,
   дальше работает шаг 1 (самосопоставление)
4. Не нашли → очередь на ручное сопоставление.
   НЕ создавать дубль молча
```

- `external_mapping` с `system` в составе ключа — не смешивать
  GUID из ЗУП и БУХ в одном поле
- **Админ-экран ручного сопоставления обязателен.** Организаций
  десятки, полная автоматика здесь опасна
- **Проверить:** бывает ли один БИН на две организации в БУХ
  (филиал, историческая копия) — тогда fallback выберет случайную

### 8.7 Договоры: BPM → БУХ

Отправка строго после финальной визы.

```json
{
  "idempotency_key": "uuid",
  "counterparty_external_id": "guid-из-бух",
  "organization_external_id": "guid",
  "project_external_id": "guid",
  "number": "...",
  "date": "2026-07-22",
  "contract_type": "...",
  "amount": 0,
  "currency": "KZT",
  "valid_from": "...",
  "valid_to": "...",
  "responsible_external_id": "guid-сотрудника",
  "bpm_document_id": "внутренний id",
  "bpm_url": "https://bpm.local/process/123",
  "approvals": [
    {
      "stage": 1,
      "position": "Руководитель проекта",
      "employee_external_id": "guid",
      "employee_name": "Иванов И.И.",
      "result": "APPROVED",
      "comment": "",
      "completed_at": "2026-07-20T14:32:00+05:00"
    }
  ]
}
```

**Нумерация договоров.** Решить до разработки: номер присваивает BPM
или БУХ. Если БУХ — BPM принимает номер в ответе и записывает себе.
Если BPM — формат не должен конфликтовать с ручным вводом бухгалтеров.
→ **уточнить**

**Сканы.** Файл хранится в BPM (S3/MinIO), в БУХ передаётся ссылка.
1С плохо переносит хранение больших объёмов бинарных данных.

### 8.8 История согласования на стороне БУХ

**Реализуются оба варианта:**

- **Копия истории** — массив `approvals` (см. выше), 1С сохраняет
  в регистр сведений. Даёт автономность, доступность в печатных формах
  и отчётах, переживает отключение BPM
- **Ссылка** `bpm_url` — реквизит договора, бухгалтер кликает и видит
  полную карточку с вложениями и перепиской

Доработка 1С: один регистр сведений + вкладка на форме договора.

**Детали:**

- Передавать **и GUID сотрудника, и ФИО строкой**: сотрудника из ЗУП
  может не быть в БУХ (разные базы). 1С ищет по GUID, не находит —
  показывает текст. История не должна ломаться из-за отсутствия ссылки
- **Момент передачи** — вместе с договором, после финальной визы.
  Промежуточные статусы не синхронизируются (это другая архитектура).
  Потребность «видеть в процессе» закрывается ссылкой
- **Переподписание** — при повторном согласовании изменённого договора
  добавляется новый блок с номером версии, старый не перезаписывается
- **Часовые пояса** — ISO 8601 со смещением (`+05:00`). 1С по умолчанию
  работает в локальном времени сервера; при выгрузке в облако легко
  получить расхождение
- **Отклонённые процессы** в БУХ не уходят вообще.
  → подтвердить с бухгалтерией, что это ожидаемо

### 9.4 Интеграция

```sql
external_mapping(
  id, system,             -- ZUP / BUH / AD
  entity_type,            -- ORGANIZATION / EMPLOYEE / ...
  internal_id, external_id, synced_at,
  UNIQUE(system, entity_type, external_id))

integration_outbox(
  id, target_system, entity_type, entity_id,
  payload JSONB, idempotency_key,
  status,                 -- PENDING / SENT / ACKED / FAILED

  created_at, sent_at)

sync_log(
  id, system, direction, entity_type,
  request JSONB, response JSONB, status, created_at)
```

`external_mapping` с `system` в ключе — потому что организация имеет
разные GUID в ЗУП и в Бухгалтерии.

## 10. Стек

| Слой | Выбор |
|---|---|
| Backend | **уточнить** (.NET / Java Spring / NestJS / Django) |
| БД | PostgreSQL |
| Очередь | RabbitMQ или Redis + воркеры |
| Frontend | React или Vue |
| Аутентификация | Keycloak (OIDC) → LDAP/AD |
| Файлы | S3-совместимое (MinIO) |
| Уведомления | SMTP + Telegram-бот + in-app |

**Готовый BPM-движок (Camunda / Flowable / Temporal) не используется.**
Логика построена на матрице, а не на нарисованных диаграммах; движок
пришлось бы использовать только как исполнитель задач либо генерировать
BPMN из матрицы. Собственный движок на 15–20 таблицах проще
и предсказуемее в поддержке.

Пересмотреть, если появятся сложные сценарии с таймерами, компенсациями
и подпроцессами.

**Конструктор матрицы — обычные таблицы с фильтрами, не BPMN-полотно.**

## 11. Уведомления

- **In-app** — колокольчик, счётчик задач, список «Мне на согласование»
- **Email** (SMTP) — адрес из AD
- **Telegram-бот** — для тех, кто не сидит в системе постоянно

Отправка **через очередь**, не синхронно с действием пользователя.

Настройки каналов и частоты в профиле пользователя — не обязательны
в v1, но поле заложить: иначе быстро придут жалобы на поток писем.

## 12. Порядок внедрения

| # | Этап | Содержание |
|---|---|---|
| 0 | **Аудит данных** | Выгрузка AD + ЗУП в CSV, замер процента сопоставления |
| 1 | Аутентификация | AD/Keycloak, синхронизация пользователей, ручной маппинг |
| 2 | Справочники | Импорт из ЗУП и БУХ, сопоставление организаций |
| 3 | Механика задач | **Служебная записка** с фиксированным маршрутом |
| 4 | Матрица | Конструктор + разрешение исполнителей + тестировщик матрицы |
| 5 | Договоры | Заявка на договор, контрагенты, outbox в БУХ |
| 6 | Обратная связь | История согласования в БУХ, роль наблюдателя, возврат из БУХ |
| 7 | Остальное | Прочие виды объектов, метрики, эскалации |

**Не начинать с заявки на договор.** Она самая сложная (двусторонний
обмен, дедупликация контрагентов), на ней легко застрять до того,
как заработает базовая механика.

**Этап 0 — до всего остального.** Сопоставление AD ↔ ЗУП почти всегда
грязнее, чем ожидается; лучше вскрыть это в начале, а не при пилоте.

## 14. Типичные ошибки (контрольный список)

- [ ] Нет версионирования матрицы → правка ломает активные процессы
- [ ] Задача привязана к пользователю, а не к должности → увольнение
      блокирует процесс
- [ ] Нет `route_snapshot` → перевод сотрудника в ЗУП меняет текущие маршруты
- [ ] Нет механизма принудительного завершения администратором
- [ ] История хранится текстом → нет аналитики по срокам
- [ ] Не продуман сценарий «документ изменили после части виз»
- [ ] Сопоставление по ФИО вместо GUID/ИИН
- [ ] Нет идемпотентности → двойники в 1С при ретраях
- [ ] Нет статуса `PENDING_EXPORT` → потерянные договоры
- [ ] Нет админ-экрана зависших обменов
- [ ] БИН сравнивается как число → потеря ведущих нулей
- [ ] Один DC в конфигурации LDAP → SPOF
- [ ] Нет Paged Results Control → тихая потеря записей при > 1000
- [ ] Роли в группах AD → зависимость от администратора домена
